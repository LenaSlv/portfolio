# CLAUDE.md

Этот файл — руководство для Claude Code при работе в этом репозитории.

## О проекте

Это не кодовый проект. Репозиторий — рабочие материалы портфолио UX/UI-дизайнера. Основной кейс: мобильное приложение заказа корпоративного транспорта для сотрудников крупного банка (iOS/Android, внутренний продукт под NDA).

Язык всех материалов — русский. Отвечать и писать тексты на русском.

## Структура

- `docs/01_кейс_финальный_текст.md` — финальный текст кейса для публикации в портфолио (структура: контекст → продуктовый контекст → роль → два сценария → решения → логика продукта)
- `docs/02_чеклист_публикации.md` — чек-лист правок макетов и страницы кейса перед публикацией, состав экранов витрины (4–5 экранов)
- `docs/03_подготовка_к_собеседованию.md` — заготовки ответов на вопросы по кейсу на собеседовании

## Критические ограничения (NDA)

- Название заказчика и продукта не раскрываются. Допустимая формулировка масштаба: «внутренний продукт для крупного банка».
- Все данные на экранах и в текстах — вымышленные: имена, номера автобусов, адреса, названия остановок, табельные номера. При редактировании не добавлять ничего похожего на реальные данные.
- В шапке кейса обязательны: заголовок, «Внутренний продукт для крупного банка · NDA», роль, платформа, пометка «экраны показаны с вымышленными данными».

## Конвенции текста кейса

- Тон: спокойный, продуктовый, без оправданий за NDA. Каждое дизайн-решение объясняется через ценность для пользователя И для бизнеса.
- Заголовки блоков — без точек в конце («Предварительный заказ», не «Предварительный заказ.»).
- Терминология едина в тексте и на макетах (пример открытого вопроса: «Табельный №» vs «Номер сотрудника» — см. чек-лист).
- «Прибытие через N мин», не «Отправление» (экран закреплённого маршрута).
- Плейсхолдеры экранов оформляются курсивом: `*[Экран: описание]*`.
- Финальная строка кейса: «Детали проекта и остальные сценарии могу показать и рассказать на встрече».

## Ключевые формулировки (не искажать при правках)

- Два сценария проектировались «как один путь»: предзаказ → закрепление за маршрутом; общие элементы (карта, карточка поездки, QR); при переходе пользователь не переучивается.
- Механика закрепления была в требованиях — не присваивать её как авторское решение. Авторское: связность двух сценариев, иерархия экрана закреплённого маршрута, структура сбора обращений («данные вместо свободного текста»).
- QR-код привязан к учётной записи и автоматически обновляется — пересланный скриншот недействителен.

## Типовые задачи

- Редактура и вычитка текста кейса (опечатки, двойные пробелы, единообразие).
- Сверка текста кейса с чек-листом публикации и заготовками к собеседованию — три документа не должны противоречить друг другу.
- При изменении фактов в `01` проверять согласованность с `02` и `03`.
